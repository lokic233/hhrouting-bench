# 06 Red Team Review - Agent F

Scope: this review attacks the benchmark plan for serving-routing mechanics only: arrivals, input/output token lengths, KV lifetime, prefix/cache reuse, sessions, tool loops, SLO regimes, and whether request-level traces can be extracted. It does not evaluate model accuracy. Source URLs are attached to each finding; if a source was gated, unclear, or not directly reusable, the finding says so.

## Executive Verdict

The plan is usable only if it sharply separates **real request traces** from **payload corpora**. Mooncake, BurstGPT, and Azure are the backbone because they expose request timing plus token counts, with Mooncake adding prefix-cache block hashes and BurstGPT adding session IDs: https://github.com/kvcache-ai/Mooncake, https://github.com/HPMLL/BurstGPT, https://github.com/Azure/AzurePublicDataset/blob/master/AzureLLMInferenceDataset2024.md.

Most other entries are not traces; they are prompt/task banks without arrival times, concurrency, cancellation, tenant IDs, or measured SLOs. Calling those "serving traces" would be rejected by a systems reviewer because the missing time dimension is exactly what routing depends on: https://huggingface.co/datasets/rungalileo/ragbench, https://huggingface.co/datasets/THUDM/LongBench-v2, https://huggingface.co/datasets/cais/mmlu.

## Top 10 Risks

1. **Overclaiming direct traces from payload-only datasets.** WildChat has conversation text and timestamps, but its card exposes top-level/turn timestamps and says `conversation_hash` is not unique; it is not a clean request-arrival trace with native token counts: https://huggingface.co/datasets/allenai/WildChat-1M. LMSYS-Chat-1M is gated and exposes conversations, not native request-arrival/token columns: https://huggingface.co/datasets/lmsys/lmsys-chat-1m.

2. **Treating accuracy benchmarks as routing benchmarks.** MMLU, MMLU-Pro, GPQA, BBH, HumanEval, MBPP, GSM8K, and MATH are mostly short single-shot prompts; they can stress batch throughput, but they do not exercise session affinity, prefix-cache reuse, multi-turn KV growth, or real arrivals: https://huggingface.co/datasets/cais/mmlu, https://huggingface.co/datasets/TIGER-Lab/MMLU-Pro, https://huggingface.co/datasets/Idavidrein/gpqa, https://huggingface.co/datasets/maveriq/bigbenchhard, https://github.com/openai/human-eval, https://huggingface.co/datasets/google-research-datasets/mbpp, https://huggingface.co/datasets/openai/gsm8k, https://huggingface.co/datasets/HuggingFaceH4/MATH-500.

3. **Unsupported token ranges are being promoted to workload definitions.** Agent B labels many ranges as estimates, including WebArena/OSWorld/WorkArena per-step input lengths, SWE agent loops, and most output lengths for coding/tool benchmarks; those should not become benchmark spec numbers until a tokenizer pass or harness trace produces measured distributions: https://github.com/web-arena-x/webarena, https://github.com/xlang-ai/OSWorld, https://github.com/ServiceNow/WorkArena, https://huggingface.co/datasets/princeton-nlp/SWE-bench_Verified.

4. **Public-access labels are inconsistent.** OpenAssistant/oasst1 is public and ungated on Hugging Face, so Agent A's `dataset_public=no` is wrong: https://huggingface.co/datasets/OpenAssistant/oasst1. MMLU is also public and ungated, so `dataset_public=no` is wrong there too: https://huggingface.co/datasets/cais/mmlu. GAIA is marked public but is gated with no-reshare terms and no license on the card: https://huggingface.co/datasets/gaia-benchmark/GAIA.

5. **License risk is underweighted.** LMSYS-Chat-1M is gated with a custom agreement and no standard license tag: https://huggingface.co/datasets/lmsys/lmsys-chat-1m. GAIA has no license on the card and adds no-reshare terms: https://huggingface.co/datasets/gaia-benchmark/GAIA. TriviaQA is tagged `license:unknown`: https://huggingface.co/datasets/mandarjoshi/trivia_qa. MultiNews is `license:other`: https://huggingface.co/datasets/alexfabbri/multi_news. The ccdv arXiv/PubMed/GovReport summarization mirrors show no license in HF metadata: https://huggingface.co/datasets/ccdv/arxiv-summarization, https://huggingface.co/datasets/ccdv/pubmed-summarization, https://huggingface.co/datasets/ccdv/govreport-summarization.

6. **ShareGPT is legally and scientifically weak as a benchmark, despite being common in vLLM-style serving tests.** The HF mirror is public and tagged Apache-2.0, but it is an anonymous scraped ShareGPT corpus with no real arrivals and altered/chunked conversations; use it only as a legacy prompt source, not as validated production behavior: https://huggingface.co/datasets/anon8231489123/ShareGPT_Vicuna_unfiltered, https://github.com/vllm-project/vllm/blob/main/benchmarks/README.md.

7. **The taxonomy is too broad in places.** `production_mixed_enterprise` mixes LLM text traces, synthetic tenant/SLO overlays, and Alibaba image-generation cluster traces; Alibaba GenTD26 may be useful for adapter/pipeline analogies, but it is not an LLM text-generation request trace: https://github.com/alibaba/clusterdata/tree/master/cluster-trace-v2026-GenAI.

8. **Browser/computer-use workloads are expensive schema sources, not cheap trace sources.** WebArena/VisualWebArena require self-hosted sites, OSWorld requires VM-style execution, and WorkArena requires a ServiceNow instance plus gated instances; the plan should not imply those can quickly yield request-level traces tonight: https://github.com/web-arena-x/webarena, https://github.com/web-arena-x/visualwebarena, https://github.com/xlang-ai/OSWorld, https://huggingface.co/datasets/xlangai/osworld_v2_tasks, https://github.com/ServiceNow/WorkArena, https://huggingface.co/datasets/ServiceNow/WorkArena-Instances.

9. **Several proposed sources are not reproducible enough for a benchmark suite.** BrowserArena has no public repo/data in the reviewed table and should remain dropped: https://github.com/web-arena-x/BrowserArena, https://arxiv.org/abs/2510.02418. SWE-Lancer's dataset hosting/gating remains insufficient evidence in the peer table and the original repo is an unstable dependency for a benchmark claim: https://github.com/openai/SWELancer-Benchmark, https://github.com/openai/preparedness. 2WikiMultihopQA relies on off-HF hosting and should be treated as a low-priority payload source: https://github.com/Alab-NII/2wikimultihop.

10. **Key routing mechanisms are missing from public data.** No reviewed public LLM trace records tenant SLO tiers, client cancellations, speculative-decoding draft/target token counts, multi-region placement, or tool-call structure in the same stream as production arrivals; Azure/BurstGPT/Mooncake cover timing/tokens/prefix/session only partially: https://github.com/Azure/AzurePublicDataset/blob/master/AzureLLMInferenceDataset2024.md, https://github.com/HPMLL/BurstGPT, https://github.com/kvcache-ai/Mooncake.

## Top 10 Fixes

1. **Rename the core split to `arrival_trace`, `payload_bank`, `schema_harness`, and `drop`.** Mooncake/BurstGPT/Azure are `arrival_trace`; RAGBench/LongBench/MMLU are `payload_bank`; WebArena/OSWorld/WorkArena are `schema_harness`: https://github.com/kvcache-ai/Mooncake, https://github.com/HPMLL/BurstGPT, https://github.com/Azure/AzurePublicDataset/blob/master/AzureLLMInferenceDataset2024.md, https://huggingface.co/datasets/rungalileo/ragbench, https://github.com/web-arena-x/webarena.

2. **Make v1 tiny and defensible.** Use Mooncake for prefix-cache routing, BurstGPT for session/burst routing, Azure 2024/2023 for arrival+token baselines, RAGBench for reusable retrieved-document prefixes, RepoBench for repo-prefix reuse, and LongBench/L-Eval for prefill stress: https://github.com/kvcache-ai/Mooncake, https://github.com/HPMLL/BurstGPT, https://github.com/Azure/AzurePublicDataset/blob/master/AzureLLMInferenceDataset2024.md, https://huggingface.co/datasets/rungalileo/ragbench, https://huggingface.co/datasets/tianyang/repobench-p, https://huggingface.co/datasets/THUDM/LongBench-v2, https://huggingface.co/datasets/L4NLP/LEval.

3. **Compute token distributions before publishing any numeric buckets.** For every payload bank with text, tokenize rows with the target tokenizer and publish p50/p90/p99 input/output lengths; do not keep char/4 or task-shape estimates as benchmark specs: https://huggingface.co/datasets/allenai/WildChat-1M, https://huggingface.co/datasets/rungalileo/ragbench, https://huggingface.co/datasets/ccdv/arxiv-summarization, https://huggingface.co/datasets/google-research-datasets/mbpp.

4. **Add a license/access gate to the benchmark manifest.** Mark LMSYS-Chat-1M, GAIA, GPQA, WorkArena instances, OSWorld v2 tasks, TriviaQA, MultiNews, and ccdv summarization mirrors as blocked/deferred until legal/access review clears them: https://huggingface.co/datasets/lmsys/lmsys-chat-1m, https://huggingface.co/datasets/gaia-benchmark/GAIA, https://huggingface.co/datasets/Idavidrein/gpqa, https://huggingface.co/datasets/ServiceNow/WorkArena-Instances, https://huggingface.co/datasets/xlangai/osworld_v2_tasks, https://huggingface.co/datasets/mandarjoshi/trivia_qa, https://huggingface.co/datasets/alexfabbri/multi_news.

5. **Downgrade batch accuracy corpora to one optional throughput class.** Keep MMLU/GSM8K/HumanEval/MBPP/BBH as a short-prompt batch stress mix, but do not give each one benchmark status for routing: https://huggingface.co/datasets/cais/mmlu, https://huggingface.co/datasets/openai/gsm8k, https://github.com/openai/human-eval, https://huggingface.co/datasets/google-research-datasets/mbpp, https://huggingface.co/datasets/maveriq/bigbenchhard.

6. **Represent synthetic overlays as explicit columns.** For every generated field, record `arrival_source`, `tenant_source`, `slo_source`, `session_source`, `prefix_source`, and `tool_loop_source`; do not let a grafted Poisson process masquerade as data from RAGBench, LongBench, or SWE-bench: https://huggingface.co/datasets/rungalileo/ragbench, https://huggingface.co/datasets/THUDM/LongBench-v2, https://huggingface.co/datasets/princeton-nlp/SWE-bench_Verified.

7. **Build one negative-control workload.** Use MMLU or HumanEval as a routing-weak workload where all requests are short, independent, and single-turn; a router should not need session or prefix logic to win there: https://huggingface.co/datasets/cais/mmlu, https://github.com/openai/human-eval.

8. **Run the agentic harnesses later, not in v1.** SWE-bench, ToolBench, AppWorld, WebArena, OSWorld, and WorkArena should enter only after the team can log per-step `(timestamp, session_id, input_tokens, output_tokens, tool_event)` from actual harness runs: https://huggingface.co/datasets/princeton-nlp/SWE-bench_Verified, https://github.com/OpenBMB/ToolBench, https://github.com/StonyBrookNLP/appworld, https://github.com/web-arena-x/webarena, https://github.com/xlang-ai/OSWorld, https://github.com/ServiceNow/WorkArena.

9. **Drop non-reproducible and non-LLM entries from the headline benchmark.** BrowserArena, Alibaba GenTD26, SWE-Lancer, and 2WikiMultihopQA can be discussed as caveats or future work, not v1 benchmark pillars: https://github.com/web-arena-x/BrowserArena, https://github.com/alibaba/clusterdata/tree/master/cluster-trace-v2026-GenAI, https://github.com/openai/SWELancer-Benchmark, https://github.com/Alab-NII/2wikimultihop.

10. **Publish validation scripts, not just a table.** A systems artifact should include scripts that re-check HF `private/gated/license/lastModified` metadata and GitHub repo existence before release; the HF API already exposes these fields for WildChat, LMSYS, GAIA, GPQA, WorkArena instances, MMLU, and OpenAssistant: https://huggingface.co/api/datasets/allenai/WildChat-1M, https://huggingface.co/api/datasets/lmsys/lmsys-chat-1m, https://huggingface.co/api/datasets/gaia-benchmark/GAIA, https://huggingface.co/api/datasets/Idavidrein/gpqa, https://huggingface.co/api/datasets/ServiceNow/WorkArena-Instances, https://huggingface.co/api/datasets/cais/mmlu, https://huggingface.co/api/datasets/OpenAssistant/oasst1.

## Benchmarks to DROP

- **BrowserArena**: no usable public repo/data in the reviewed evidence; keep as "not reproducible offline": https://github.com/web-arena-x/BrowserArena, https://arxiv.org/abs/2510.02418.
- **Alibaba GenAI GenTD26 as an LLM benchmark**: image-generation serving traces are not text LLM request traces; at most use as a non-LLM adapter/pipeline analogy: https://github.com/alibaba/clusterdata/tree/master/cluster-trace-v2026-GenAI.
- **SWE-Lancer for v1**: dataset publicness and hosting remain insufficient evidence; do not base a v1 benchmark on it: https://github.com/openai/SWELancer-Benchmark, https://github.com/openai/preparedness.
- **GAIA and GPQA for routing v1**: gated and anti-leakage terms make them poor serving-routing sources, and they are mainly accuracy/agent-eval tasks unless a harness trace is generated: https://huggingface.co/datasets/gaia-benchmark/GAIA, https://huggingface.co/datasets/Idavidrein/gpqa.
- **ShareGPT as a validated trace**: keep only as a legacy prompt corpus; drop any claim that it represents production arrivals or intact sessions: https://huggingface.co/datasets/anon8231489123/ShareGPT_Vicuna_unfiltered, https://github.com/vllm-project/vllm/blob/main/benchmarks/README.md.
- **UltraChat as real chat traffic**: synthetic ChatGPT-generated dialogs can be a stress corpus, not evidence of human arrival/session distributions: https://huggingface.co/datasets/HuggingFaceH4/ultrachat_200k, https://huggingface.co/datasets/openbmb/UltraChat.
- **2WikiMultihopQA for v1**: data access is less clean than HF-hosted RAGBench/HotpotQA/Qasper, and it adds little routing coverage beyond multi-hop QA payloads: https://github.com/Alab-NII/2wikimultihop, https://huggingface.co/datasets/rungalileo/ragbench, https://huggingface.co/datasets/hotpotqa/hotpot_qa, https://huggingface.co/datasets/allenai/qasper.
- **TriviaQA until license review clears it**: HF metadata says `license:unknown`; do not ship it as a clean public benchmark source: https://huggingface.co/datasets/mandarjoshi/trivia_qa.
- **MultiNews until license review clears it**: HF metadata says `license:other`, and Agent A notes non-commercial usage terms: https://huggingface.co/datasets/alexfabbri/multi_news.
- **MATH-500 as a license-clean source**: HF metadata for this mirror shows no license in the API output, so use original-source licensing or a different math payload if licensing matters: https://huggingface.co/datasets/HuggingFaceH4/MATH-500.

## Benchmarks to PRIORITIZE

- **Mooncake Trace**: highest-priority prefix-cache workload because it includes `timestamp`, `input_length`, `output_length`, and `hash_ids`: https://github.com/kvcache-ai/Mooncake, https://www.usenix.org/system/files/fast25-qin.pdf.
- **BurstGPT**: highest-priority session/burst workload because it includes timestamps, session IDs, request/response token counts, model, elapsed time, and log type: https://github.com/HPMLL/BurstGPT, https://arxiv.org/abs/2401.17644.
- **Azure LLM Inference Trace 2024/2023**: highest-priority simple arrival+token baselines because the trace pages describe timestamp, context-token, and generated-token fields without prompt text: https://github.com/Azure/AzurePublicDataset/blob/master/AzureLLMInferenceDataset2024.md, https://github.com/Azure/AzurePublicDataset/blob/master/AzureLLMInferenceDataset2023.md.
- **RAGBench**: good payload bank for repeated retrieved-document prefixes; use synthesized arrivals and label them as such: https://huggingface.co/datasets/rungalileo/ragbench.
- **RepoBench**: good payload bank for repo/file prefix reuse and strict autocomplete-style prefill-to-decode imbalance; license/access must use the exact HF config selected: https://huggingface.co/datasets/tianyang/repobench-p, https://huggingface.co/datasets/tianyang/repobench_python_v1.1.
- **LongBench v2 and L-Eval**: good extreme-prefill payload banks, with short outputs and no native arrivals: https://huggingface.co/datasets/THUDM/LongBench-v2, https://huggingface.co/datasets/L4NLP/LEval.
- **arXiv/PubMed summarization only after license review**: useful long-input/long-output payloads, but the ccdv HF metadata shows no license in API output: https://huggingface.co/datasets/ccdv/arxiv-summarization, https://huggingface.co/datasets/ccdv/pubmed-summarization.
- **ToolBench**: useful for static tool-call trajectories, but not as a production arrival source: https://github.com/OpenBMB/ToolBench.
- **AppWorld**: useful for tool/API loop counts and task metadata, but traces need harness execution: https://github.com/StonyBrookNLP/appworld, https://huggingface.co/datasets/LukaszTP/AppWorld-Tasks.
- **WebArena text-only variant**: useful later for browser-step prefill/action imbalance; defer VM/multimodal variants until harness logging exists: https://github.com/web-arena-x/webarena, https://github.com/web-arena-x/visualwebarena.

## Claims We Should AVOID Making

- Avoid: "HHRouting-Bench is based on public request-level traces across many domains." Safer wording: only Mooncake, BurstGPT, and Azure-style sources are clean request-level arrival+token traces; most other sources are payload banks or harness schemas: https://github.com/kvcache-ai/Mooncake, https://github.com/HPMLL/BurstGPT, https://github.com/Azure/AzurePublicDataset/blob/master/AzureLLMInferenceDataset2024.md.
- Avoid: "WildChat/LMSYS provide production arrival traces." Safer wording: WildChat/LMSYS provide public or gated conversation corpora with timestamps/conversation fields, but request arrivals and native token counts require derivation or are absent: https://huggingface.co/datasets/allenai/WildChat-1M, https://huggingface.co/datasets/lmsys/lmsys-chat-1m.
- Avoid: "ShareGPT is a validated serving trace." Safer wording: ShareGPT is a common prompt corpus for serving benchmarks that synthesize arrivals externally: https://huggingface.co/datasets/anon8231489123/ShareGPT_Vicuna_unfiltered, https://github.com/vllm-project/vllm/blob/main/benchmarks/README.md.
- Avoid: "LongBench v2 proves real long-context serving behavior." Safer wording: LongBench v2 supplies long-context payloads; arrival/concurrency/SLO behavior must be synthesized: https://huggingface.co/datasets/THUDM/LongBench-v2.
- Avoid: "ToolBench/AppWorld/WebArena are routing traces." Safer wording: they expose tool/browser task structure and need harness-generated per-step serving traces: https://github.com/OpenBMB/ToolBench, https://github.com/StonyBrookNLP/appworld, https://github.com/web-arena-x/webarena.
- Avoid: "MMLU/GPQA/BBH materially evaluate serving routers." Safer wording: they are short-prompt throughput controls and mostly lack routing-specific dimensions: https://huggingface.co/datasets/cais/mmlu, https://huggingface.co/datasets/Idavidrein/gpqa, https://huggingface.co/datasets/maveriq/bigbenchhard.
- Avoid: "All benchmark licenses are research-safe." Safer wording: LMSYS, GAIA, GPQA, TriviaQA, MultiNews, ccdv summarization mirrors, and WorkArena instances need explicit access/license handling: https://huggingface.co/datasets/lmsys/lmsys-chat-1m, https://huggingface.co/datasets/gaia-benchmark/GAIA, https://huggingface.co/datasets/Idavidrein/gpqa, https://huggingface.co/datasets/mandarjoshi/trivia_qa, https://huggingface.co/datasets/alexfabbri/multi_news, https://huggingface.co/datasets/ccdv/arxiv-summarization, https://huggingface.co/datasets/ServiceNow/WorkArena-Instances.
- Avoid: "The benchmark covers speculative decoding, cancellation, tenant tiers, or multi-region routing." Safer wording: those dimensions require synthetic overlays because the reviewed traces do not expose those fields: https://github.com/Azure/AzurePublicDataset/blob/master/AzureLLMInferenceDataset2024.md, https://github.com/HPMLL/BurstGPT, https://github.com/kvcache-ai/Mooncake.
- Avoid: "BrowserArena is part of the benchmark." Safer wording: BrowserArena is excluded until public code/data exists: https://github.com/web-arena-x/BrowserArena, https://arxiv.org/abs/2510.02418.
- Avoid: "Estimated token ranges are measured distributions." Safer wording: only native token columns or post-tokenized payloads should be called measured; char/4 and task-shape ranges are estimates: https://github.com/Azure/AzurePublicDataset/blob/master/AzureLLMInferenceDataset2024.md, https://github.com/HPMLL/BurstGPT, https://github.com/kvcache-ai/Mooncake.

## Claims We Can SAFELY Make

- HHRouting-Bench can use Mooncake as a prefix-cache-aware trace source because the reviewed source exposes request timing, input/output lengths, and cache hash IDs: https://github.com/kvcache-ai/Mooncake, https://www.usenix.org/system/files/fast25-qin.pdf.
- HHRouting-Bench can use BurstGPT as a session-aware arrival/token trace source because the source exposes timestamps, session IDs, and request/response token fields: https://github.com/HPMLL/BurstGPT, https://arxiv.org/abs/2401.17644.
- HHRouting-Bench can use Azure LLM Inference 2024/2023 as clean arrival+token baselines with no prompt text, because the source pages describe timestamp/context-token/generated-token fields: https://github.com/Azure/AzurePublicDataset/blob/master/AzureLLMInferenceDataset2024.md, https://github.com/Azure/AzurePublicDataset/blob/master/AzureLLMInferenceDataset2023.md.
- HHRouting-Bench can use RAGBench, LongBench v2, L-Eval, RepoBench, and summarization corpora as payload banks, provided arrivals and SLOs are explicitly synthetic: https://huggingface.co/datasets/rungalileo/ragbench, https://huggingface.co/datasets/THUDM/LongBench-v2, https://huggingface.co/datasets/L4NLP/LEval, https://huggingface.co/datasets/tianyang/repobench-p, https://huggingface.co/datasets/ccdv/arxiv-summarization.
- HHRouting-Bench can include one short-prompt batch-throughput control using MMLU/GSM8K/HumanEval/MBPP, provided the paper says this class has little session/prefix/tool routing value: https://huggingface.co/datasets/cais/mmlu, https://huggingface.co/datasets/openai/gsm8k, https://github.com/openai/human-eval, https://huggingface.co/datasets/google-research-datasets/mbpp.
- ToolBench/AppWorld/WebArena can inform future tool/browser-loop traces, but only after an execution harness logs per-step token lengths and timestamps: https://github.com/OpenBMB/ToolBench, https://github.com/StonyBrookNLP/appworld, https://github.com/web-arena-x/webarena.
- WildChat is a public ODC-BY chat corpus with conversation text, timestamps, hashed IP, request headers, and non-unique conversation hashes, but token counts and true request arrivals require derivation/caveats: https://huggingface.co/datasets/allenai/WildChat-1M.
- LMSYS-Chat-1M is gated and governed by a custom dataset agreement, so it should be optional or deferred rather than a required public benchmark dependency: https://huggingface.co/datasets/lmsys/lmsys-chat-1m.
- OpenAssistant is public and Apache-2.0, but its tree structure requires path flattening before it can become a linear serving workload: https://huggingface.co/datasets/OpenAssistant/oasst1.
- The benchmark will look serious if it reports negative findings and exclusions, including BrowserArena, gated/unclear-license sources, and pure accuracy tasks with little routing value: https://github.com/web-arena-x/BrowserArena, https://huggingface.co/datasets/gaia-benchmark/GAIA, https://huggingface.co/datasets/cais/mmlu.

## What Would Make HHRouting-Bench Look UNSERIOUS

- Publishing a table where MMLU, GPQA, BBH, HumanEval, and MBPP look as important for routing as Mooncake/BurstGPT/Azure; these are short single-shot eval corpora, not routing traces: https://huggingface.co/datasets/cais/mmlu, https://huggingface.co/datasets/Idavidrein/gpqa, https://huggingface.co/datasets/maveriq/bigbenchhard, https://github.com/openai/human-eval, https://huggingface.co/datasets/google-research-datasets/mbpp, https://github.com/kvcache-ai/Mooncake, https://github.com/HPMLL/BurstGPT.
- Using "realistic arrivals" for datasets that have no timestamps and then hiding that arrivals are synthetic; RAGBench, LongBench v2, SWE-bench Verified, and MMLU need explicit synthetic-arrival labels: https://huggingface.co/datasets/rungalileo/ragbench, https://huggingface.co/datasets/THUDM/LongBench-v2, https://huggingface.co/datasets/princeton-nlp/SWE-bench_Verified, https://huggingface.co/datasets/cais/mmlu.
- Treating license/gating as paperwork instead of benchmark reproducibility; GAIA, GPQA, WorkArena instances, LMSYS-Chat-1M, TriviaQA, and MultiNews are reproducibility risks without clear access/legal handling: https://huggingface.co/datasets/gaia-benchmark/GAIA, https://huggingface.co/datasets/Idavidrein/gpqa, https://huggingface.co/datasets/ServiceNow/WorkArena-Instances, https://huggingface.co/datasets/lmsys/lmsys-chat-1m, https://huggingface.co/datasets/mandarjoshi/trivia_qa, https://huggingface.co/datasets/alexfabbri/multi_news.
- Presenting hand-estimated token ranges as measured p99 tails; the trace sources with native token columns are different from text payload banks that need post-tokenization: https://github.com/Azure/AzurePublicDataset/blob/master/AzureLLMInferenceDataset2024.md, https://github.com/HPMLL/BurstGPT, https://github.com/kvcache-ai/Mooncake, https://huggingface.co/datasets/allenai/WildChat-1M.
- Including BrowserArena or Alibaba GenTD26 in the headline LLM-serving benchmark without strong caveats; BrowserArena lacks public reproducible artifacts in the reviewed evidence, and Alibaba GenTD26 is image-generation serving rather than LLM text generation: https://github.com/web-arena-x/BrowserArena, https://arxiv.org/abs/2510.02418, https://github.com/alibaba/clusterdata/tree/master/cluster-trace-v2026-GenAI.

## Final Recommendation

Ship a v1 benchmark story with three layers: **real arrival/token traces** from Mooncake, BurstGPT, and Azure; **payload banks** from RAGBench, RepoBench, LongBench/L-Eval, and selected summarization data after license review; and **future harness traces** from ToolBench/AppWorld/WebArena/SWE-bench after actual execution logging. Anything else should be a drop, a negative finding, or a clearly synthetic overlay: https://github.com/kvcache-ai/Mooncake, https://github.com/HPMLL/BurstGPT, https://github.com/Azure/AzurePublicDataset/blob/master/AzureLLMInferenceDataset2024.md, https://huggingface.co/datasets/rungalileo/ragbench, https://huggingface.co/datasets/tianyang/repobench-p, https://huggingface.co/datasets/THUDM/LongBench-v2, https://huggingface.co/datasets/L4NLP/LEval, https://github.com/OpenBMB/ToolBench, https://github.com/StonyBrookNLP/appworld, https://github.com/web-arena-x/webarena, https://huggingface.co/datasets/princeton-nlp/SWE-bench_Verified.
